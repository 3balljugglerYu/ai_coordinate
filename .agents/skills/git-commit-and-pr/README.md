# git-commit-and-pr スキル

## このスキルは何をするものか
`git-commit-and-pr` は、次の5つを支援するスキルです。

- Conventional Commits 形式に沿ったコミットメッセージの作成・改善
- 「プッシュして」時の `add -> commit -> push` 実行
- チームのPRテンプレートに沿った、テスト計画付きのPR説明文の下書き
- 必要時は認証確認から `commit -> push -> PR作成` までの実行
- 「マージしました」後の安全な後処理（`main` 最新化とローカルブランチ削除）

ユーザーが「PRを作る」「PR説明を用意する」「コミットメッセージを整える」といった作業を行うときに使います。  
特に「プッシュして」といった依頼は、プッシュモードで扱います。
特に「PRを作成して」「PR作って」といった依頼は、原則として実行モードで扱います。
また「マージしました」といった依頼では、後処理モード（`main` へ移動・pull・条件付きブランチ削除）で扱います。

## 実行モードの要点
- GitHubアクセス可否を確認してから進行します（このプロジェクトでは `.local/github-auth.env` を優先）。
- `main` ブランチ上ならそのまま進まず、推奨ブランチ名を複数提示して選択してもらいます。
- 選択後、該当ブランチで commit / push / PR作成を実行します。
- 追加確認が必要なときは、推奨案を先頭にした選択肢形式で質問します。

## プッシュモード（「プッシュして」）
- 実行内容: `add -> commit -> push`
- デフォルト:
  1. `add` は staged のみ（`git add -A` は自動実行しない）
  2. `main` / `master` にいる場合は停止し、ブランチ選択を要求
- このリポジトリでは `scripts/git-commit-and-push.sh` を優先利用

## マージ後の後処理モード
- トリガー: 「マージしました」
- 実行順:
  1. 現在ブランチ（移動元）を記録
  2. `main` に移動
  3. `git pull --ff-only origin main`
  4. リモートの移動元ブランチが削除済みか確認
  5. 削除済みならローカルブランチを `git branch -d` で削除
- 重要ルール:
  - リモートブランチが未削除なら、その時点で終了
  - 強制削除（`-D`）は実行しない
  - このリポジトリでは `scripts/post-merge-cleanup.sh` を優先利用

## SKILL.md の日本語訳

### 目的
このスキルはエージェントが以下を行うためのものです。

- Conventional Commits 仕様に従ったGitコミットメッセージを提案し、必要に応じて改善する
- チームのPRテンプレートに沿って、明確なテスト計画を含むPR説明文を作成する

### Conventional Commits
コミットメッセージを生成するときは、常に以下に従います。

- 仕様: <https://www.conventionalcommits.org/en/v1.0.0/>
- 基本形式:

```text
type(optional-scope): short, imperative summary
```

よく使う type（例）:

- `feat`: 新機能
- `fix`: バグ修正
- `docs`: ドキュメント変更
- `refactor`: バグ修正や機能追加を伴わないコード改善
- `test`: テストの追加・更新
- `chore`: メンテナンス作業、ビルドスクリプト変更など

このプロジェクトでのルール:

- 可能な限り意味のある scope（コンポーネント名、モジュール名、APIパスなど）を付ける
- summary は短く、命令形で書く  
  （例: `add user deactivate API`。`added` や `adding` は避ける）
- 論理的に別の変更が混在している場合は、複数コミットへの分割を提案する

### PRテンプレート
PR説明文は次の構造で作成します。

```markdown
### 概要
[{Jira ticket number if included, can be multiple}]
{description}

### 変更内容
{what is updated}

### テスト方法
{how to run the test, show what test has been added both runn or unit test}
```

ガイドライン:

- 「概要」には以下を含める
  - 関連Jiraチケット番号（`[]` で囲む）
  - 変更の意図を短く要約した説明
- 「変更内容」には具体的なコード変更を記載する
  - 新規エンドポイント、挙動変更、リファクタリング、設定変更など
- 「テスト方法」では以下を明確にする
  - 変更確認手順を具体的に記載する
  - 可能なら runn シナリオと unit test の両方に触れる
  - 必要に応じて実行コマンド例（例: `cargo make internal-tests-runn "..."`）を示す

### エージェントの利用手順
1. コード変更の準備ができたら:
- 差分を自然言語で要約する
- その要約をもとに Conventional Commits 候補を1つ以上提案する
- 最終確定前に、コミットメッセージの確認または調整をユーザーに依頼する

2. ユーザーがPR準備をしているとき:
- 次の情報を確認する
  - 関連Jiraチケット番号（あれば）
  - 変更の目的を表す短い説明
  - 主な変更点と影響範囲
  - 追加・更新したテスト（runn / unit test）
- 上記テンプレートに沿ってPR説明文を作成する

3. テスト欄の扱い:
- runn テストと unit/integration テストを明確に分ける
  - runn: ファイルパスと実行方法
  - unit / integration: 実行コマンド（例: `cargo test`）
- 不足しているテストがある場合は、追加すべきテストを提案する

4. 常に守ること:
- コミットメッセージとPR説明文の整合性を保つ
- コード中の用語・API名と同じ表現を使う
- レビュアーが影響範囲を理解できる具体性を保ちつつ、簡潔に書く
