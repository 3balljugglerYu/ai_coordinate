# ユーザーストーリー

英語版: [`user-stories.en.md`](./user-stories.en.md)

- 最終更新日: `2026-06-16`
- 想定読者: このリポジトリで作業する開発者、プロダクトオーナー、デザイナー
- 役割: 現行の主要ユーザージャーニーを説明する正本

## このドキュメントの役割

このドキュメントは、現行アプリでユーザーがどのような目的で画面を行き来し、どの時点で認証、投稿、通知、ペルコイン購入が発生するかを、ストーリー単位で把握するための正本です。

厳密な仕様は `docs/product/requirements.md`、画面遷移は `docs/product/screen-flow.md`、データと RPC は `docs/architecture/data.md` を参照してください。

推奨の読み順:

1. このファイルで主要体験をつかむ
2. `docs/product/screen-flow.md` で route と戻り先を確認する
3. `docs/product/requirements.md` で詳細仕様を確認する
4. `docs/architecture/data.md` でデータ保存、RLS、RPC を確認する

## スコープ

このドキュメントは、現行の主要なユーザー向け導線に絞っています。

- 対象: ホーム閲覧、登録、生成(コーディネート / One-Tap Style)、投稿、コミュニティ参加、コレクション、Inspire / Creator Looks、絵師カタログ、マイページ、ミッション、通知・運営アナウンス、ペルコイン購入
- 非対象: 管理画面の詳細運用、将来構想のみの機能、実装されていない UI 案

## Story 1: ゲストが作品を見て、初回生成のために登録する

### 目的

未認証ユーザーがホームで作品を眺め、興味を持ったあとに登録して生成体験へ進む。

### 主な route

- `/`
- `/posts/[id]`
- `/login`
- `/signup`
- `/coordinate`

### ストーリー

1. ユーザーはホームを開き、投稿一覧を閲覧する。
2. 気になる投稿を見つけたら投稿詳細を開き、作品や投稿者を確認する。
3. 生成を試したくなり、主要ナビゲーションの「コーディネート」を押す。
4. 未認証のため、ログインまたは新規登録へ誘導される。
5. ユーザーはメール認証または OAuth で登録を完了し、生成画面へ戻る。

### 期待される結果

- ホームと公開投稿は未認証でも閲覧できる。
- 認証必須機能に入る時点で、ログインまたは新規登録が求められる。
- 登録完了後、生成の文脈を失わずに `/coordinate` へ進める。

## Story 2: 新規ユーザーがチュートリアルを経て最初の画像を生成する

### 目的

新規ユーザーが初回ボーナスとチュートリアルを使って、最初の画像生成を完了する。

### 主な route

- `/coordinate`
- `/challenge`
- `/my-page`

### ストーリー

1. 新規登録直後、ユーザーには初期のペルコイン残高が付与される。
2. ユーザーは `/coordinate` で人物画像、プロンプト、背景変更有無、生成枚数を入力する。
3. 対象ユーザーにはチュートリアル導線が表示され、完了時に追加ボーナスを受け取れる。
4. 画面には、選択したモデルと枚数に応じた必要ペルコイン数が表示される。
5. ユーザーが生成を開始すると、画像は 1 枚ずつ順に生成される。
6. ペルコインは開始時ではなく、生成に成功した枚数分だけ消費される。
7. 生成済み画像は保存され、ユーザーはプレビューや拡大表示で結果を確認する。

### 期待される結果

- 新規登録ボーナスだけで最初の体験が可能である。
- 生成失敗時は成功分のみ消費される。
- 生成結果はあとで投稿しなくてもマイページから見返せる。

## Story 3: ユーザーが生成画像を投稿し、コミュニティで反応を受ける

### 目的

ユーザーが生成画像を公開し、ホーム、投稿詳細、通知を通じてコミュニティ参加を始める。

### 主な route

- `/coordinate`
- `/my-page`
- `/posts/[id]`
- `/users/[userId]`
- `/notifications`

### ストーリー

1. ユーザーは生成完了後のプレビュー、またはマイページ上の未投稿画像から投稿導線を開く。
2. 投稿モーダルで必要に応じてキャプションを入力し、投稿を完了する。
3. 投稿後の画像はホーム一覧に反映される。
4. 他の認証済みユーザーは、ホームの「新着」「オススメ」「フォロー」導線や投稿詳細から作品を見つける。
5. 作品に対して、いいね、コメント、フォローなどのアクションが行われる。
6. 投稿者は `/notifications` で反応を確認し、通知から投稿詳細や相手プロフィールへ遷移する。

### 期待される結果

- 投稿は生成フローのすぐ後でも、マイページから後日でも行える。
- コミュニティ反応は通知に集約され、通知から適切な詳細画面へ戻れる。
- ソーシャル操作は認証済みユーザーに限定される。

## Story 4: ユーザーがマイページで作品とアカウントを管理する

### 目的

ユーザーが自分の作品、プロフィール、残高、アカウント設定を一箇所で管理する。

### 主な route

- `/my-page`
- `/my-page/[id]`
- `/posts/[id]?from=my-page`
- `/my-page/credits`
- `/my-page/account`
- `/my-page/contact`
- `/account/reactivate`

### ストーリー

1. ユーザーは `/my-page` でプロフィール、統計、生成画像一覧、ペルコイン残高を確認する。
2. 画像一覧から、共有投稿詳細または所有者向けの詳細画面へ進む。
3. 投稿済み画像はキャプション編集や投稿取り消しを行える。
4. 未投稿画像は必要に応じて投稿、または削除を行える。
5. ユーザーは `/my-page/credits` で残高と取引履歴を確認する。
6. `/my-page/account` ではアカウント管理、通報履歴、ブロック、退会関連の操作を行う。
7. 退会予約後に復帰する場合は `/account/reactivate` を使う。

### 期待される結果

- 生成画像の公開状態に応じて、できる操作が変わる。
- マイページからの遷移では `from=my-page` により戻り先が保たれる。
- 残高確認、取引履歴確認、購入導線がマイページ系 route に集約されている。

## Story 5: ユーザーがミッションとボーナスで継続利用する

### 目的

ユーザーがチュートリアル、連続ログイン、紹介、日次投稿などの導線を通じて継続利用する。

### 主な route

- `/challenge`
- `/coordinate`
- `/my-page`

### ストーリー

1. ユーザーは `/challenge` を開き、現在のボーナス導線や進行状況を確認する。
2. 初回導線ではチュートリアル完了によるボーナス取得が継続率向上に使われる。
3. 日次の利用では、連続ログインボーナスや日次投稿ボーナスがペルコイン再獲得の動機になる。
4. 紹介コード経由の登録が成立すると、紹介者側に紹介ボーナスが付与される。
5. ユーザーは受け取ったボーナスを使って、再び画像生成や投稿へ戻る。

### 期待される結果

- ボーナス導線は、単独機能ではなく生成と投稿の再訪を促す役割を持つ。
- 継続利用のインセンティブは `/challenge` と各行動完了時の付与に分かれている。
- 付与済みペルコインは残高と取引履歴に反映される。

## Story 6: ユーザーがペルコインを購入して生成を継続する

### 目的

ユーザーが残高不足時に購入へ進み、決済完了後すぐ生成を再開できる。

### 主な route

- `/my-page/credits`
- `/my-page/credits/purchase`
- `/coordinate`

### ストーリー

1. ユーザーはマイページの残高カード、または生成中の不足案内から購入導線へ進む。
2. `/my-page/credits/purchase` で Stripe Pricing Table からパッケージを選択する。
3. 現行パッケージは 110 / 240 / 960 / 1,900 / 4,800 ペルコインである。
4. Checkout 完了後、Webhook により残高と取引履歴が更新される。
5. ユーザーは購入後に残高を確認し、そのまま生成画面へ戻って再開する。

### 期待される結果

- 残高不足は購入導線へ自然につながる。
- 決済反映は取引履歴でも確認できる。
- 現行パッケージや単価の正本は `docs/business/monetization.md` と実装ファイルにある。

## Story 7: ゲスト/ユーザーが One-Tap Style でプリセットから手早く着せ替える

### 目的

ユーザーが細かい入力をせず、用意されたスタイルプリセットを選ぶだけで着せ替え画像を生成する。未ログインでもお試し生成を体験できる。

### 主な route

- `/style`
- `/coordinate`

### ストーリー

1. ユーザーは生成画面で「コーディネート」と「One-Tap Style」のモード切替タブから One-Tap Style を選ぶ。
2. カテゴリ別に並ぶスタイルプリセットから 1 つを選び、人物画像をアップロードまたは保存済み画像から選択する。
3. カテゴリによっては背景変更・生成モデル・プロンプト入力の有無が切り替わる。
4. 未ログインのゲストは「お試し生成」として 1 日 1 回まで無料で生成できる(`coordinate` と枠を共有)。
5. ログイン済みユーザーは選択モデルに応じたペルコインを消費して生成し、結果ギャラリーで確認する。
6. 一部プリセットは段階解放(一定回数の生成でカテゴリ内のプリセットが順次開放)の対象になる。

### 期待される結果

- プリセット選択だけで生成でき、`/coordinate` の自由入力フローより手数が少ない。
- ゲストのお試し生成は 1 日 1 回に制限され、登録導線へ自然につながる。
- 認証済みの生成はモデル別ペルコインを消費し、残高・取引履歴に反映される。

## Story 8: ユーザーがコレクションを集めて台紙をコンプリートし、シェアする

### 目的

ユーザーがテーマ別コレクション(神コレ/ぷち神)のスタイルを集め、台紙(ボード)を完成させて達成感を得て、SNS でシェアする。

### 主な route

- `/style`
- `/collections/wafer`
- `/m/[token]`

### ストーリー

1. ユーザーは One-Tap Style で、コレクション対象のスタイルプリセットを生成して 1 体ずつ集める。
2. コレクション進捗モーダルで、現在の収集数(N 体中いくつ)と収集済みシールを確認する。
3. プリセットは段階解放され、生成を重ねるごとに次のスタイルが開放される。
4. 全 N 体を集めると、完成演出(紙吹雪)とともに台紙作成導線が表示される。
5. ユーザーが台紙を作成すると、収集画像を合成した 1 枚の台紙画像が生成される。
6. 完成した台紙は `/m/[token]` の公開ページからシェアでき、OGP 画像付きで X 等に共有される。
7. 神コレを完走したユーザーには、続編の「ぷち神」コレクションが解放される。

### 期待される結果

- コレクション機能はログイン必須で、進捗・完成はアカウントに紐づく。
- 完走はリテンション施策として段階解放・続編解放につながる。
- 完成した台紙は公開シェアリンクから誰でも閲覧でき、新規ユーザーの流入導線になる。

## Story 9: ユーザーが Inspire / Creator Looks のテンプレートを使う・投稿する

### 目的

ユーザーが他ユーザーの投稿したスタイルテンプレートを使って生成し、また自作テンプレートを投稿してコミュニティに共有する。

### 主な route

- `/inspire/[templateId]`
- `/inspire/submit`
- `/coordinate`

### ストーリー

1. ユーザーは公開中のスタイルテンプレート(ホームや生成画面のカルーセル等で露出)から気になるものを選ぶ。
2. テンプレート詳細でクリエイター情報・利用数を確認し、人物画像を入れて生成する。
3. 生成結果はギャラリーに表示され、`/coordinate` の詳細へ展開して活用できる。
4. テンプレートを投稿したいユーザーは `/inspire/submit` から画像とタイトルを登録する。
5. Creator Looks 投稿では、投稿元(自作のみ)と複数の同意事項(著作権・第三者 IP・二次利用等)への同意が求められる。
6. 投稿は draft → pending(運営審査)→ visible の順で公開される。

### 期待される結果

- テンプレートの利用・投稿はログイン必須。投稿は運営審査を経て公開される。
- Creator Looks 投稿には同意取得と投稿数上限(最大 5 件)が設けられている。
- テンプレート画像は署名付き URL 経由で配信される(プロンプトは非公開)。

## Story 10: ユーザーが絵師カタログを閲覧し、作品掲載を申請する

### 目的

ユーザー(および X の絵師)が企画ごとにまとめられた作品集を没入リーダーで閲覧し、自作の掲載を申請する。

### 主な route

- `/catalog`
- `/catalog/[slug]`
- `/catalog/submit`

### ストーリー

1. ユーザーは `/catalog` で企画別のカタログ一覧を見る(未ログインでも閲覧可)。
2. カタログを開くと、ページめくり演出の没入リーダー(本形式)で作品を 1 ページずつ閲覧する。
3. 各ページから作者の X アカウントや元ツイートへ遷移できる。
4. 絵師は `/catalog/submit` から、作品画像・X アカウント・元ツイート URL 等を申請する(未ログインでも申請可)。
5. 申請は status: pending として登録され、運営が元ツイートで本人性を確認して承認/却下する。
6. 承認された作品は公開カタログのリーダーに表示順で掲載される。

### 期待される結果

- カタログの閲覧・申請は未ログインでも可能で、生成・投稿機能とは独立している。
- 申請は運営審査を経て公開され、本人性は元ツイート URL で確認される。

## Story 11: ユーザーが運営アナウンスを受け取り、確認する

### 目的

ユーザーが運営からの公式アナウンスを通知画面で受け取り、内容を確認する。

### 主な route

- `/notifications`
- `/notifications/announcements/[id]`

### ストーリー

1. ユーザーは `/notifications` の「お知らせ」タブで、運営が公開したアナウンス一覧を見る。
2. 未読アナウンスにはバッジ(ドット)が表示される。
3. アナウンスを開くと `/notifications/announcements/[id]` で本文(リッチテキスト・画像)を読む。
4. 閲覧すると自動で既読になり、未読バッジが消える。

### 期待される結果

- アナウンスは運営(管理者)のみが作成・公開でき、公開済みのものだけがユーザーに表示される。
- 既読/未読が管理され、コミュニティ反応の通知とは別タブで集約される。

## 補助メモ

- ホームの主導線は「新着」「オススメ」「フォロー」を前提に考える。
- `daily / week / month` の集計概念は実装上存在しても、現行の主要体験説明では中心に置かない。
- 購入パッケージやボーナス量は変わりうるため、数値の正本は必ず `docs/business/monetization.md` と対応する実装を参照する。
- 画面遷移の戻り先は `docs/product/screen-flow.md` の `from` クエリルールを優先する。
